# Capability Health Contract: collab-wiki

_Generated by Wave 5 to satisfy RT-1 (Degradation is concretely defined per capability with measurable detection signals)._

The manifold commits B1 — all four primary capabilities are equal INVARIANTs. That promise is only testable if "degraded" has a concrete definition per capability. This document provides that definition, with detection signals and alert thresholds, so operators can answer the question "is the system working?" unambiguously.

---

## Capability 1: CRDT Multi-User Collaboration

**Constraints:** U1 (presence), U2 (read-only on disconnect), U3 (two-state indicator), T1 (ephemeral CRDT), T2 (p95 < 300ms peer latency), O8 (3s/100op flush), RT-2 (binding).

### Working state

- Every open document has an active `crdt.Room` in memory.
- A new op applied by any editor is broadcast to all peers of that room within 300ms p95 (`OpAckedEvent`).
- Ops are flushed to disk within the O8 window (≤3s idle or ≤100 ops per room), producing `OpSavedEvent`.
- The client UI distinguishes **synced** (ack received) from **saved** (disk-durable) states per TN6.

### Degraded state (any of the following)

1. Active CRDT room count drops to zero unexpectedly while `wiki.Hub.Subscribe` has active subscribers.
2. `OpAckedEvent` delivery latency p95 exceeds 300ms over a 5-minute window.
3. `OpSavedEvent` stops emitting for any room with buffered ops older than the O8 idle window (flush stalled).
4. `crdt.InProcBroadcaster.DroppedFor(roomID)` increases monotonically for >1 minute (slow subscribers causing persistent drops).

### Detection signals

| Signal | Source | Threshold |
|---|---|---|
| `crdt_op_broadcast_latency_seconds{quantile=0.95}` | Prometheus (Wave-5 wiring) | ≤ 0.3 |
| `crdt_op_flush_age_seconds` (oldest unflushed op per room) | Prometheus gauge | ≤ 3.0 |
| `crdt_op_dropped_total{room}` rate of change | Prometheus counter | = 0 over 60s |
| `crdt_rooms_active` | Prometheus gauge | ≥ active-subscriber count |

### Alert

- **Warn**: any degradation condition persisting > 2 minutes
- **Critical**: `op_flush_age_seconds` > 10s for any room (bounded-loss window breach)

## Capability 2: API Editing (API-KEY auth)

**Constraints:** S1 (dual principal), S5 (body cap), S6 (media quota), TN8 (unified RBAC).

### Working state

- Requests bearing `API-KEY: <token>` with a valid token authenticate as either a user PAT (acts as user) or a workspace service account (distinct principal).
- Authentication produces a `principal.Principal` with `PrincipalType = "service_account"` or `"user"` and a workspace role.
- Edits via API are authorized through the same RBAC function as cookie-authenticated users.
- Every API write produces an audit event with `principal_type` + `principal_id`.

### Degraded state

1. Valid `API-KEY` requests are rejected with 401.
2. `API-KEY` requests authenticate but produce a Principal with empty `WorkspaceID` (authz silently blocks all actions).
3. Write operations via API succeed without a corresponding audit event (S2 regression).

### Detection signals

| Signal | Source | Threshold |
|---|---|---|
| `api_auth_failures_total` rate | Prometheus counter | stable (no sudden spikes vs 24h baseline) |
| `audit_events_total{action="doc.edit",principal_type="service_account"}` | Prometheus counter | > 0 when API writes are observed |
| `api_request_duration_seconds{path="/api/pages/",quantile=0.95}` | Prometheus histogram | ≤ 0.1 |

### Alert

- **Warn**: API-KEY 401 rate exceeds 1/min for 5 minutes (possibly rotated key not deployed; possibly attack).
- **Critical**: API write events exist but matching audit events are missing for > 60s (S2 regression).

## Capability 3: Password + OIDC Authentication

**Constraints:** S3 (argon2id + CSRF), S4 (OIDC JWKS/iss/aud), S8 (claim mapping schema versioning), RT-10 (auth RL floor), RT-11 (OIDC adapter).

### Working state

- `/api/login` accepts valid credentials and issues a Secure + HttpOnly + SameSite=Lax cookie.
- OIDC adapter (when configured) validates id_token signatures, enforces `iss`/`aud`, rejects expired/`alg: none` tokens, and maps claims per the configured `ClaimMapping`.
- `/api/login` + `/api/auth/*` are rate-limited per-IP with the always-on floor (RT-10).
- The server-side limiter honors X-Forwarded-For ONLY from trusted-proxy CIDRs.

### Degraded state

1. Valid credentials fail login with 5xx (server-side issue, not wrong password).
2. OIDC id_tokens with invalid signatures OR unexpected `iss` OR expired `exp` are accepted as valid.
3. `auth.login_failed` rate exceeds 10x the 24h baseline and rate-limit `OnDeny` counter does NOT increase proportionally (RL floor not firing).
4. OIDC adapter `ClaimMapping.SchemaVersion` mismatch is not surfaced as a config error (silent misattribution).

### Detection signals

| Signal | Source | Threshold |
|---|---|---|
| `auth_login_success_total` rate | Prometheus | non-zero during business hours |
| `auth_login_failed_total` rate | Prometheus | ≤ 10× baseline |
| `ratelimit_denied_total{path="/api/login"}` | Prometheus counter | tracks failed-login rate |
| `oidc_signature_verification_failures_total` | Prometheus counter | > 0 indicates attacks OR config drift |

### Alert

- **Warn**: login 5xx rate > 0.1%.
- **Critical**: OIDC signature-failure rate spike > 100/min (attack) OR JWKS cache refresh failing > 5 minutes (IdP unreachable, rotation-blocked).

## Capability 4: Git Sync (push-only)

**Constraints:** T6 (push-only), T7 (one workspace = one repo), T10 (bounded push queue), O7 (snapshot backup), O9 (repo size alerts), RT-7 (push journal), RT-12 (bounded drain).

### Working state

- Every workspace's configured git remote receives pushed commits on the configured cadence.
- Push queue depth stays bounded (≤ 100 pending commits per workspace under normal load).
- SIGTERM → Server.Serve graceful shutdown drains the push queue within the configured window (30s default); any overflow spills to `internal/journal` for startup replay.
- On startup, `journal.Replay` runs before `/readyz` returns 200 — no ready state with unreplayed commits.
- `O7` periodic snapshot completes within its configured interval (1h default), covering both markdown and blob-store bytes.

### Degraded state

1. Git push backlog exceeds 100 queued commits for any workspace.
2. Git remote unreachable for > 5 minutes (pushes stall).
3. Journal file exists with unreplayed entries after `NewServer` completion but `/readyz` returns 200 (RT-7.2 regression).
4. Snapshot cycle fails 3 consecutive times.
5. Repo size exceeds 5GB (O9 alert threshold) — likely media-in-git regression (should live in blob store, TN4).

### Detection signals

| Signal | Source | Threshold |
|---|---|---|
| `git_push_queue_depth` per workspace | Prometheus gauge | ≤ 100 |
| `git_push_last_success_timestamp_seconds` age | Prometheus gauge | ≤ 600 seconds |
| `journal_entries_pending` | Prometheus gauge | = 0 after startup |
| `snapshot_last_success_timestamp_seconds` age | Prometheus gauge | ≤ 2× snapshot interval |
| `git_repo_size_bytes` | Prometheus gauge | ≤ 5GB (O9 alert); warn at 1GB |

### Alert

- **Warn**: push queue > 100 OR snapshot lag > 2h OR repo size > 1GB.
- **Critical**: push queue > 1000 (T10 cap) OR `/readyz` is green but `journal_entries_pending` > 0 (invariant violation) OR repo > 5GB.

## Cross-cutting observability

These apply across all four capabilities:

| Signal | Source | Notes |
|---|---|---|
| `up{job="phronesis"}` | Prometheus service-discovery | Basic liveness |
| `process_resident_memory_bytes` | Prometheus | memory-leak detection |
| `audit_events_dropped_total` | Prometheus counter | S9 bounded-loss signal |
| `/api/readyz` status code | `http_probe` | Readiness: composes data-dir + drainer + journal |

## Revision trigger

This contract is revised whenever:
- A new capability is added to B1 (outcome expands).
- A detection signal's source changes (e.g., moving to OpenTelemetry metrics).
- A TIGHTENED propagation from a future tension resolution changes a threshold.
